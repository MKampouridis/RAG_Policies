## Stack (current production)
- **Retrieval:** Chroma (dense, `nomic-embed-text`) + BM25 (`rank_bm25`) fused with **RRF**, then **ColBERT reranking** (`GTE-ModernColBERT` via pylate, MaxSim). Over-fetch pool → rerank top-6. `is_current` archive pre-filter + family-recency dedupe.
- **Generation:** local Ollama. Query **contextualizer** = `qwen2.5:7b-instruct`; **answer generator** = `gemma3:12b` (changed from the 14B I reported in round 5 — see §1); **judge** (eval only) = `qwen2.5:14b-instruct`, with `phi4` as a neutral cross-family judge.
- **Corpus:** ~12.6k chunks over hundreds of Essex PDFs. The hard problem is unchanged: many documents are **near-duplicate "siblings"** (same RoA boilerplate, differing only by programme / degree-length / cohort-year / partner-institution). Retrieval failures are almost always "retrieved the WRONG sibling," not "off-topic."
- **Eval:** 40 questions × 2 turns = 80 turns, plus an independent 40-question holdout set, deterministic (`RAG_DETERMINISTIC=1`). Metrics: strict hit@6, **evidence-sufficient@6** (headline), answer_score (1–5 judge), keyphrase_coverage, and **groundedness / faithfulness-to-context** (the hallucination metric).

## What I did since round 5, and what I found

**1. Generator bake-off (10 local models) — and it OVERTURNED my round-5 generator choice.** Round 5 I switched production to the local 14B on a "bigger = better grounding" finding. I then bake-off'd 10 models on **identical fixed contexts** (retrieval held constant, only the generator varies), judged groundedness with the 14B, then re-judged with a **neutral `phi4`** judge to strip self-judging bias. Result: **`gemma3:12b` is the best production pick, and I switched to it.** The decisive finding is the **hit/miss split**: gemma3 matches the field on retrieval-HIT turns (answer_score ~3.94 vs 14B 4.12) but on retrieval-MISS turns it **faithfully abstains** (~92% grounded under neutral judge) while the 14B **guesses from parametric memory** (~85% grounded → hallucinates a figure). This **directly solves the round-5 "stronger models are confidently wrong on misses" problem** — it turned out to be *qwen-line-specific*, not a law; gemma3/gpt-oss break the trend. So **generator *choice* is itself the abstention mechanism** the round-4 gate diagnostic couldn't build — a model that naturally says "not found" on weak context, no gate required. (Methodology note I want critiqued: a single judge badly overstated the margin — gemma3-as-judge is too lenient at 100% self-score, qwen2.5:14b is harsh, `phi4` was the usable neutral one; gpt-oss:20b's apparent lead was a qwen-judge artifact that vanished under phi4. Never trust a single judge for close calls.)

**2. Retrieval bake-off — I finally gave retrieval the same rigor as the generator, and it CLOSED the frontier.** Decomposed into a staged pipeline so recall vs ranking failures separate:
- **Recall diagnostic (both sets): 81% of RoA misses are RANKING failures** — the gold doc IS in the candidate pool, mis-ranked below a sibling. Only 19% are recall failures (gold absent). → the reranker, not the embedder, is the high-leverage lever.
- **Reranker sweep:** every cross-encoder (bge-reranker-v2-m3/base, mxbai) caps at **~+3 family-hit and is holdout-unstable**; the current ColBERT is already top of the pack. Identity-salient passage formatting: **marginal (+2)**.
- **The capstone — can REASONING break the sibling tie that SCORING can't?** On exactly the ranking-failure turns, I gave a capable LLM the query + the distinct competing document identities from the pool and asked it to pick the right sibling. **Result: 3/22 (14%)** — above random (~2.5% for the 17–64-option sets) but nowhere near reliable, and it rubber-stamps shared vocabulary (e.g. picks a doc about *uncapped* marks for a *capped*-mark question). **Neither scoring, formatting, nor reasoning disambiguates the siblings, because the distinguishing fact is not in the query.** This is the rigorously-earned confirmation: **the residual RoA misses are an underspecification problem, not a retrieval problem.** I did NOT run an embedder bake-off — it only targets the 19% recall tail and can't answer a query that never names its programme.

**3. Structured parameter-extraction ("Class F", ambiguity→enumeration) — FALSIFIED.** The proposal: for an underspecified query, don't retrieve one sibling — extract the parameter across ALL programme docs and enumerate `{programme → value}`. Piloted by extracting Merit/Distinction thresholds from 40 current PGT RoA docs (gemma3): extraction was **reliable (85%)** but the values are **perfectly uniform** (every programme's Merit = 60, Distinction = 70). So the "ambiguity" was never real — any retrieved sibling already answers correctly (a gold-multiplicity artifact). Enumeration would emit "60, 60, 60…". General kill: uniform parameters make enumeration pointless; varying parameters either name the programme (not underspecified) or produce a table that's worse UX than just asking.

**4. Contextualizer bake-off + "unified model" idea — production choice validated, unification REJECTED.** The follow-up query contextualizer is the one place that model touches retrieval, so I bake-off'd it (follow-up hit@6 only): `qwen2.5:7b` (prod) 80.0%, `qwen2.5:14b` 82.5% (+2.5, not worth 2× cost), **`gemma3:12b` 73.8% (−6.2 overall, −15pp RoA)**. So the tempting "use one model for both contextualize + generate to halve RAM" idea is **dead** — the generator is a *worse* contextualizer; faithful query-rewriting is a narrow instruct task the small qwen2.5:7b wins. `qwen3:8b` disqualified on latency (ignores `/no_think`, ~140s/turn thinking bloat on a synchronous pre-retrieval call).

**5. Metric fix (a round-4 caveat, now closed).** evidence-sufficient@6 used brittle exact-substring keyphrase matching. I built a **judge-based reference-answer-containment** check for the flagged turns: it recovers ~+6 genuinely-under-credited turns (evidence-sufficient@6 ≈ **91% overall / 97.5% policy / ~85% RoA**), and — importantly — it separates true metric artifacts from **genuine** retrieval misses (5 turns stay missed even under a deliberately lenient judge). So the residual is real, not a scoring illusion.

**6. Also:** routing pre-validation (learned facet routing) — **rejected without a build**, already falsified 3× and 81% of misses don't name the facet to route on. Model housekeeping done (5 models kept). Production runs 3 distinct models (gemma3 generate + qwen2.5:7b contextualize + nomic embed ≈ 13GB) → needs `OLLAMA_MAX_LOADED_MODELS ≥ 2`.

## Already exhaustively tried and FALSIFIED — please don't re-suggest these
Cross-encoder rerankers (multiple), ColBERT as first-stage retriever, **more/other rerankers in a formal sweep (all cap ~+3, holdout-unstable)**, **LLM-reasoning reranking / disambiguation (14% ≈ near-random)**, **identity-salient rerank passages**, alternate embedding models (mxbai, bge-m3), embedding ensembles, SPLADE, weighted score fusion (vs RRF), hard + soft facet filtering, **learned/macro document-routing**, pseudo-query (HyDE) expansion, CRAG verification, selective multi-hop decomposition, header-weight boosting, wider rerank pools, inline per-claim citations (regressed groundedness), verbatim-figure prompt rules, **structured parameter-enumeration (Class F)**, **unified contextualize+generate model**, **retrieval-confidence abstention gates** (no signal separates hit from miss — the wrong sibling is as confident/on-topic as the right doc). ~30 architecture experiments total; the only net-positive *retrieval* gains ever came from **data/lexical hygiene**, not architecture. The two net-positive *generation* levers were **generator choice** (capability on hits, faithful abstention on misses) and **chunk size**.
